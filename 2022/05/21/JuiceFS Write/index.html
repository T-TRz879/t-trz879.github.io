Write相关类和接口

​ 最顶层是一个DataWriter，它维护着一个由文件Inode为key，fileWriter为value的map集合，这样可以更好的管理所有的文件。
​ 每个fileWriter中都会有一个chuks，其是由Chunk自增编号「文件写入的offset / 64MB」为key，chunkWriter为value的map集合，和DataWriter维护fileWriter一样的逻辑。
​ 每个chunkWriter有slices列表，其中记录着当前Chunk的所有sliceWriter。
​ 一个sliceWriter中有一个Writer接口，一个Writer被wChunk继承，wChunk中存放了一个Page二维数组，用来存储sliceWriter中的data数据，将其放入page中等待其它线程调用Flush方法将Page中数据写入Blcok中。

Write流程

Kernel到JuiceFS流程

​ JuiceFS将一个文件最终拆分成多个4MB的Block存储在磁盘中。基础流程大概如下：首先会经过Kernel Fuse的Wirte接口。此接口会将用户写入的文件每次以128KB大小的数据流调用JuiceFS Client的Write接口，JuiceFS Client的Write接口会将每次128KB数据缓存在内存中，等达到Block阈值4MB时，会将一个Block数据写入到磁盘中。

  1. Application将数据以1MB以128KB每片的形式将其发送到Kernel提供的接口
  2. Kernel将每个128KB的数据传送到User Space的JuiceFS Clinet中
  3. JuiceFS Client将Kernel传来的数据放入Slice中，一个Slice中有一个或多个Block
  4. JuiceFS Client等待一个Block放满后，或者一个Chunk放满「64MB」后会将其写入具体的存储介质中「上传对象存储/保存在本地磁盘」

各种Write流程

Write方法调用流程

  1. File和VFS中都有相应的write接口，前者是相对对象存储相关的，而后者是文件挂载相关的。通过得知file的信息，如Inode、位置、名称等，会创建一个fileWriter对象将当前file信息放入其中，当前file的操作都会调用fileWriter对象
  2. fileWriter.write()中根据off计算当前数据需要写入第几个Chunk中「off/64MB」和在Chunk中哪个位置开始写入「off%64MB」
  3. fileWriter.writeChunk()中根据indx查询具体Chunk，Chunk存在，则直接获取；反之需根据传入indx创建当前Chunk。根据返回的chunkWriter对象，在其中寻找具体需写入且已存在的sliceWriter对象，不存在则返回空
  4. 通过sliceWriter调用write写入具体的slice
  5. wChunk的writeAt方法会将需写入的数据放入具体的page中「wChunk使用的Page二维数组中」

Write主要代码

fileWriter.Write

  1. 判断fileWriter中chunkWriterMap中的所有sliceWriter数量是否大于1000，大于1000则让其等一会

  2. 判断dataWriter中使用的内存空间是否已经大于系统配置的BufferSize，大于则进入循环等待

  3. fileWriter的其它线程正在flush，当前线程需要等待。如果等待超时且用户取消操作则取消本次write

  4. 通过off计算出indx「off / 64MB」和pos「off % 64MB」，可得知在key为indx的Chunk中从其Chunk的下标为pos的位置开始写入data个数据

  5. 对data数据进行处理，但其中会出现如下两种情况。「假设当前data中有100长度的数据」

    • [0,99]的数据位于同一个Chunk
    • [0,40]位于第一个Chunk，[41,99]位于第二个Chunk

  6. 利用go对数据的切片操作可对data.len进行循环，每次结束将当前剩余的数据切出来，用作下一次循环

    1. 如果当前pos + data数据大于64MB，本次循环只处理长度为n「64MB - pos」的数据
    2. 调用fileWriter.writeChunk方法，将data的[0,n - 1]的数据写入
    3. 如果data还有数据，那么就需要将indx++，且更新pos「pos + n % 64MB」的位置，进入下一次循环

  7. 更新fileWriter中目前已经写入的数据长度

// off是当前文件的需要从off开始写入data中的数据
func (f *fileWriter) Write(ctx meta.Context, off uint64, data []byte) syscall.Errno {
    for {
    // 当前fileWriter中所有的silcesWriter数量 >= 1000 让其等一会
        if f.totalSlices() < 1000 {
            break
        }
        time.Sleep(time.Millisecond)
    }
  // 判断当前已经使用过的内存和实际内存的关系
    if f.w.usedBufferSize() > f.w.bufferSize {
        // slow down
        time.Sleep(time.Millisecond * 10)
        for f.w.usedBufferSize() > f.w.bufferSize*2 {
            time.Sleep(time.Millisecond * 100)
        }
    }

    s := time.Now()
    f.Lock()
    defer f.Unlock() 
  // data中的数据大小
    size := uint64(len(data))
    f.writewaiting++
  // 写入等待超时
    for f.flushwaiting > 0 {
        if f.writecond.WaitWithTimeout(time.Second) && ctx.Canceled() {
            f.writewaiting--
            logger.Warnf("write %d interrupted after %d", f.inode, time.Since(s))
            return syscall.EINTR
        }
    }
    f.writewaiting--
    // indx代表写入第几个Chunk
  // pos代表从第indx的Chunk的下标pos开始写
  // meta.ChunkSize = 64MB
    indx := uint32(off / meta.ChunkSize)
    pos := uint32(off % meta.ChunkSize)
  // 将本次的data数据处理完，可能出现跨Chunk 最多执行两次
    for len(data) > 0 {
        n := uint32(len(data))
        if pos+n > meta.ChunkSize {
            // 如果 pos + n > 64MB 本次只把 [0,64MB - pos]大小的内容写入到indx的Chunk中
            n = meta.ChunkSize - pos
        }
        // data[:n] 取得[0,n - 1]的数据
        if st := f.writeChunk(ctx, indx, pos, data[:n]); st != 0 {
            return st
        }
        // 下一次需将data[n:]写入Chunk
        data = data[n:]
        indx++
        // 更新写入(实际n个)数据后的pos,只有两种可能 小于64Mb或等于64MB
        pos = (pos + n) % meta.ChunkSize
    }
    if off+size > f.length {
        f.length = off + size
    }
    return f.err
}

fileWriter.writeChunk

  1. 通过indx寻找Chunk，如果当前Chunk不存在则创建
  2. 通过Chunk寻找当前off在Chunk的哪个sliceWriter中
  3. 如果sliceWriter未找到，则需要创建新的sliceWriter
  4. 使用当前Chunk写入数据长度传入sliceWriter.write方法
// indx指写入第几个Chunk off是当前Chunk需要从off开始写入data个数据 这里的off[0,64MB)
func (f *fileWriter) writeChunk(ctx meta.Context, indx uint32, off uint32, data []byte) syscall.Errno {
    // 寻找fileWriter中indx的chunkWriter
    c := f.findChunk(indx)
    // 在上一步「找到/创建的」chunkWriter中寻找sliceWriter
    s := c.findWritableSlice(off, uint32(len(data)))
  // 不存在就创建一个新的sliceWriter对象
    if s == nil {
        s = &sliceWriter{
            chunk:   c,
            off:     off,	// 当前Chunk中的已经写入的索引
            writer:  f.w.store.NewWriter(0),
            notify:  utils.NewCond(&f.Mutex),
            started: time.Now(),
        }
        c.slices = append(c.slices, s)
        if len(c.slices) == 1 {
            f.w.Lock()
            f.refs++
            f.w.Unlock()
      // 将chunkWriter中的sliceWriter列表按顺序写入Block
            go c.commitThread()
        }
    }
    // off-s.off代表 当前写入数据索引 与 当前Chunk在文件开始索引 的差值
    // off是当前Chunk的写入位置 s.off是当前slice在Chunk中的开始位置 一旦slice创建s.off就固定了
    // 所以要用off - s.off得到当前slice的写入位置
    return s.write(ctx, off-s.off, data)
}

fileWriter.findChunk

// i = indx指该写入的指定Chunk
func (f *fileWriter) findChunk(i uint32) *chunkWriter {
   // 如果f.chunks[i] 不存在则初始化
   c := f.chunks[i]
   if c == nil {
      c = &chunkWriter{indx: i, file: f}
      f.chunks[i] = c
   }
   return c
}

fileWriter.findWritableSlice

  1. 对当前chunkWriter中的slices进行遍历，判断是否有pos所在的slice

  2. 对slices进行倒序遍历方便顺序写只需一次就可找到目标slice

  3. flushoff有如下两种情况

    • flushoff为BlockSize的整数倍
    • flushoff不为BlockSize的整数倍

  4. flushoff计算出当前正在写入的slice中具体Block的开始索引「s.off+flushoff」和结束索引「s.off+s.slen」，该操作可以对其进行覆盖写

  5. 从后往前寻找超过4次并没有寻找到具体的slice，则需要将slice进行flush操作

// pos是该Chunk需要从pos开始写入size个数据 这里的off[0,64MB)
func (c *chunkWriter) findWritableSlice(pos uint32, size uint32) *sliceWriter { 
  // blockSize为4MB
   blockSize := uint32(c.file.w.blockSize)
  // 如果c.slices不存在则在会创建具体的slice
   for i := range c.slices {s
     // 从后往前寻找pos所在的slice
      s := c.slices[len(c.slices)-1-i]
      if !s.freezed {
         flushoff := s.slen / blockSize * blockSize
         if pos >= s.off+flushoff && pos <= s.off+s.slen {
            return s
         } else if i > 3 {
            // 当前s不可有内容写入，需要对其进行flush
            s.freezed = true
            go s.flushData()
         }
      }
      if pos < s.off+s.slen && s.off < pos+size {
         // overlaped 这块JuiceFS没有实现
         // TODO: write into multiple slices
         return nil
      }
   }
   return nil
}

sliceWriter.write

  1. 调用WriteAt具体方法将data写入pages中，写入pages后对s.slen进行更新
  2. s.slen达到64MB时调flushData
  3. s.slen达到4MB时调用FulshTo方法
  4. off小于4MB时准备ChunkId
func (s *sliceWriter) write(ctx meta.Context, off uint32, data []uint8) syscall.Errno {
    f := s.chunk.file
    // 不是真正将数据写入文件中,只是处理当前data中的数据改放入pages中
    _, err := s.writer.WriteAt(data, int64(off))
    if err != nil {
        logger.Warnf("write: chunk: %d off: %d %s", s.id, off, err)
        return syscall.EIO
    }
  // 更新s.slen
    if off+uint32(len(data)) > s.slen {
        s.slen = off + uint32(len(data))
    }
    s.lastMod = time.Now()
    if s.slen == meta.ChunkSize {
        // s.slen达到一个ChunkSize 64MB
    s.freezed = true
        go s.flushData()
    } else if int(s.slen) >= f.w.blockSize {
    // s.slen大于BlockSize 4MB
        if s.id > 0 {
      // 及时进行FlushTo
            err := s.writer.FlushTo(int(s.slen))
            if err != nil {
                logger.Warnf("write: chunk: %d off: %d %s", s.id, off, err)
                return syscall.EIO
            }
        } else if int(off) <= f.w.blockSize {
            // 当off小于4MB时，就代表当前是一个新Chunk还未分配ChunkId，现在分配一个ChunkId，方便后续使用这个Id
            // ChunkId也不是一定会在这里创建 在FlushData需要使用chunkId的地方会调用此方法
            go s.prepareID(ctx, false)
        }
    }
    return 0
}

wChunk.WriteAt

方法中参数说明

参数 说明
int64 off 当前数据载Chunk中的写入位置
int indx 当前数据应写入第几个Block
int boff 当前数据在Block中写入的位置
int bs 当前每页的大小，有两种情况「64KB和4MB」
int bi 当前数据在Block的哪个Page中写入
int bo 当前数据在Block的具体Page的哪个索引开始写入
c.pages 以Page为基础的 16 * 64的二维数组，总大小上限64MB，单个一维为4MB
int left 本需需要写入数据，有两种情况「left + n <= len(p) 和 left + n > len(p)」
int n 每次循环写入page中的数据

Page二维数据说明

  1. Page是一个16 * 64或16 * 1的二维数组，16 * 64因为每个Chunk在indx为0的时候「刚开始写入数据还没有到达4MB」。16 * 1因为每个Chunk在indx = 0写完后发现还有数据，那么就不会对二维继续拆分，直接将其设置为一个Block

WriteAt流程

  1. 当前off+len(p) > 64MB，说明当前p应该写入下一个Chunk中
  2. 如果off < c.uploaded，说明p已经写入，无需继续写入
  3. Fill previous blocks with zeros是多线程写入时，后面片段先于前面片段执行，则需要用空block填充
  4. indx将数据写入哪个Block「也就是哪一个 c.pages[?] 中」，boff是将数据在具体Block中的哪个下标开始写入
  5. bs默认是64KB，当indx > 0时就将其设置为4MB
  6. bi是当前数据需要写入Block「c.pages[?]」的哪个page中[c.pages[indx] [?]]，bo是当前数据在page「c.pages[indx] [?]」中的具体哪个位置开始写入
  7. 如果bi < 当前c.pages[indx]的长度，则直接从c.pages中获取page。否则创建一个page将其放入到c.pages中
  8. 当前写入的数据可能涉及到跨Block，这个和上面跨Chunk的道理一样
  9. 将p中数据copy到pages中
func (c *wChunk) WriteAt(p []byte, off int64) (n int, err error) {
   // off + p.len > 64MB return error
   if int(off)+len(p) > chunkSize {
      return 0, fmt.Errorf("write out of chunk boudary: %d > %d", int(off)+len(p), chunkSize)
   } // cannot overwrite exist block
   if off < int64(c.uploaded) {
      return 0, fmt.Errorf("Cannot overwrite uploaded block: %d < %d", off, c.uploaded)
   }

   // Fill previous blocks with zeros 这是随机写的一个策略 将之前没有写入内容的地方用zeros填充
   if c.length < int(off) {
      zeros := make([]byte, int(off)-c.length)
      _, _ = c.WriteAt(zeros, int64(c.length))
   }
   // 45[chunkId]_0[blockId]_4194304[blockSize] c.id = chunkId blockId是通过indx计算出来的
   for n < len(p) { 
      // indx represent  (off + n) / 4MB
      indx := c.index(int(off) + n)
      // boff represent current block off
      // boff也是计算出当前Chunk已经写入到哪里了,下面会通过boff和4MB来计算出[]当前的数据写入哪个Block
      boff := (int(off) + n) % c.store.conf.BlockSize
      // pageSize = 64KB
      var bs = pageSize
      // BlockSize is 4MB
      if indx > 0 || bs > c.store.conf.BlockSize {
         bs = c.store.conf.BlockSize
      }
      // bi 是当前Block中第几个位置
      bi := boff / bs
      // bo 是当前Block中第几个位置的索引开始写入
      bo := boff % bs
      var page *Page
      if bi < len(c.pages[indx]) {
         page = c.pages[indx][bi]
      } else {
         page = allocPage(bs)
         page.Data = page.Data[:0]
         c.pages[indx] = append(c.pages[indx], page)
      }
      left := len(p) - n
      // 下面对本个block的操作 和 chunk操作一样,都是判断是否超过4MB
      if bo+left > bs {
         page.Data = page.Data[:bs]
      } else if len(page.Data) < bo+left {
         page.Data = page.Data[:bo+left]
      } // 将p中内容copy到page.Data中
      n += copy(page.Data[bo:], p[n:])
   }
   if int(off)+n > c.length {
      //更新c.length长度,用来write和read时知道目前已经写了多少内容
      c.length = int(off) + n
   }
   return n, nil
}

Flush相关方法

dataWriter.flushAll

// 创建dataWriter时候,会启用一个线程一直去调用flushAll
func (w *dataWriter) flushAll() {
   for {
      w.Lock()
      now := time.Now()
      for _, f := range w.files {
        // 使用files时，将其使用数 + 1
         f.refs++
         w.Unlock()
         tooMany := f.totalSlices() > 800
         f.Lock()

         lastBit := uint32(rand.Int() % 2) // choose half of chunks randomly
         for i, c := range f.chunks {
            hs := len(c.slices) / 2
            for j, s := range c.slices {
               // 当前时间 - sliceWriter开始时间 > 5s 或 当前时间 - sliceWriter修改时间 > 1s 
              // 或 当前总共slice > 800 且 佛系的把数据写入block 且 [0,len(c.slices) / 2)
               if !s.freezed && (now.Sub(s.started) > flushDuration || now.Sub(s.lastMod) > time.Second ||
                  tooMany && i%2 == lastBit && j <= hs) {
                  s.freezed = true
                  go s.flushData()
               }
            }
         }
         f.Unlock()
         w.free(f)
         w.Lock()
      }
      w.Unlock()
      time.Sleep(time.Millisecond * 100)
   }
}

sliceWriter.flushData

// freezed, no more data
func (s *sliceWriter) flushData() {
   defer s.markDone()
   if s.slen == 0 {
      return
   }
   // 准备chunkId
   s.prepareID(meta.Background, true)
   if s.err != 0 {
      logger.Infof("flush inode:%d chunk: %s", s.chunk.file.inode, s.err)
      s.writer.Abort()
      return
   }
   s.length = s.slen
   if err := s.writer.Finish(int(s.length)); err != nil {
      logger.Errorf("upload chunk %v (length: %v) fail: %s", s.id, s.length, err)
      s.writer.Abort()
      s.err = syscall.EIO
   }
   s.writer = nil
}

sliceWriter.prepareID

func (s *sliceWriter) prepareID(ctx meta.Context, retry bool) {
   f := s.chunk.file
   f.Lock()
   for s.id == 0 {
      var id uint64
      f.Unlock()
      // 根据当前的nextId和maxId来判断是否需要更新maxId
      // 获取当前的自增nextId
      st := f.w.m.NewChunk(ctx, &id)
      f.Lock()
      if st != 0 && st != syscall.EIO {
         s.err = st
         break
      }
      if !retry || st == 0 {
         if s.id == 0 {
            s.id = id
         }
         break
      }
      f.Unlock()
      logger.Debugf("meta is not available: %s", st)
      time.Sleep(time.Millisecond * 100)
      f.Lock()
   }
   if s.writer != nil && s.writer.ID() == 0 {
      s.writer.SetID(s.id)
   }
   logger.Warnf("Set ChunkId:%d", s.id)
   f.Unlock()
}

}

baseMeta.NewChunk

func (m *baseMeta) NewChunk(ctx Context, chunkid *uint64) syscall.Errno {
   m.freeMu.Lock()
   defer m.freeMu.Unlock()
   if m.freeChunks.next >= m.freeChunks.maxid {
      // 从redis中nextChunk且更这个值
      v, err := m.en.incrCounter("nextChunk", chunkIDBatch)
      if err != nil {
         return errno(err)
      }
      m.freeChunks.next = uint64(v) - chunkIDBatch
      m.freeChunks.maxid = uint64(v)
   }
   // 将id赋值且自增
   *chunkid = m.freeChunks.next
   m.freeChunks.next++
   return 0
}

注意事项

编辑和拷贝的异同

相同点：

  • 编辑和拷贝都会调用Write方法，调用write方法流程都是一样

不同点：

  • 编辑文件保存后，会调用fuse接口以8KB每份的大小将数据传入Write方法中；复制文件，会调用fuse接口以128KB每份的大小将文件复制
  • 编辑文件保存后，一个chunkId的所有Block数量不会超过6份；复制文件，一个chunkId的所有Block数量不会超过16份

Block数量不同的原因：

  • 复制和编辑单次写入的大小不一，Flush将Page数据处理进Block时，不光判断当前slice写入数据长度满足4MB或64MB，还会如下几个点进行考察
    • 现在时间 - sliceWriter创建时间 > 5s
    • 现在时间 - 上次修改时间 > 1s
    • 当前DataWriter持有的fileWriterMap中所有的slice是否大于800个 且 当前slice列表前一半 且 一个1/2的取模「50%的可能性」
  • 和运行JuiceFS机器的处理性能也有很大关系，相同的时间内性能越好的机器处理的数据越多，则写入Slice的数据也更多，在Flush时单个Slice转换为Block的数量必然会增多「其实讨论这个是没什么必要的，作者只是想拿出来简单引出JuiceFS Flush的巧妙之处」

编辑

​ 编辑文件保存时，write每次是8KB，off逐渐递增8KB

​ 编辑文件保存存储Block时，每个chunkId的Block数量不会超过6个

复制

​ 复制文件时，write每次是128KB，off随着每次写入递增128KB

​ 复制文件保存存储Block时，每个chunkId的Block数量不会超过16个
